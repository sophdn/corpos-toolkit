# Single-source the work-action param shape (chain `single-source-action-describe`)

Status: **T1 decision recorded** (audit + direction call; no production code).

## Problem

Two independently hand-maintained systems each describe every `work`-surface
action's parameter shape, and they have drifted:

| System | Source | Consumed by | Availability |
| --- | --- | --- | --- |
| **actiondocs** | `blueprints/action-docs/work/*.toml` → parsed by `go/internal/actiondocs` | `admin.action_describe` | **gated** on `--action-docs-dir` (describe is disabled when the corpus is absent) |
| **actionSpecs** | `actionSpecs` literal in `go/internal/work/actions_discovery.go` | `work_actions` and the T4 param-error renderer (`CallShape` / `IdentifierRequiredError` in `go/internal/work/param_errors.go`) | **always** compiled into the binary |

`task_reopen` was the reported case: the TOML declares a single **required**
`slug` and no `id`; `actionSpecs` declares `id` / `slug` / `chain_slug` all
optional (`id` preferred). An agent that reads `action_describe` and one that
trips the param-error renderer get contradictory call shapes.

## Audit finding: the drift is structural, not incidental — **38 of 42** in-both actions disagree

Both systems list params, but with **different modelling conventions**, so the
divergence is pervasive. (Generated by a throwaway diagnostic that loaded both
sources and diffed the overlap fields; the diagnostic was removed — T1 ships no
code.)

Presence: `actionSpecs` lists 49 actions, the work TOML corpus 42. The 7
spec-only actions (`arc_review_audit`, `emit_commit_landed`,
`pending_decisions_claim`, `task_unstart`, `trained_model_list/promote/retire`)
have **no TOML chunk at all** — `action_describe` can't describe them. No
TOML-only actions.

Drift classes across the 42 in-both actions:

1. **Example presence (~35 actions).** `actionSpecs` carries an `Example` on
   nearly every action; the TOML carries one on almost none (only `forge` has a
   TOML example, where the spec lacks one).
2. **identify-by-id-or-slug, modelled wrong in TOML (~17 actions:** `bug_read`,
   `bug_reopen`, `bug_resolve`, `bug_stamp_sha`, `suggestion_{read,reopen,resolve,stamp_sha}`,
   `task_{blockers,cancel,complete,read,reopen,stamp_sha,start}`, `task_unblock`**).**
   TOML marks `slug` **required** and omits `id`; the handler actually accepts
   `id` OR `slug` (both optional, one-of). `actionSpecs` models this correctly.
   This is the `task_reopen` class.
3. **Alias modelling mismatch.** `actionSpecs` lists every accepted key
   (canonical + alias) as a flat sibling param, marking alias-ness only in prose
   ("Alias of commit_sha"). The TOML separates canonical `[[params]]` from
   structured `[[param_aliases]]` (`from`→`to`). So `bug_resolve` shows
   `notes`/`resolution_summary`/`summary`/`sha` as spec params but TOML aliases;
   `chain_close/state/status` even disagree on the **canonical name**
   (TOML `chain` + alias `slug`; spec `slug`); `chain_find` (spec `pattern` /
   TOML `query`); `task_block`/`task_unblock` (TOML canonical `blocked_by` —
   the prior silently-dropped-param bug — vs spec `blocker_slug`).
4. **Genuine TOML type errors.** `roadmap_set` / `roadmap_preview_set` `items`
   typed `string` in TOML, `object[]` in spec (it's a list);
   `review_arc_for_filing` `triggers` `optional_string` vs `string[]`,
   `user_turns_since_review` `optional_string` vs `int64`. The existing
   `param_type_parity` gate skips these because the handler field is non-primitive.
5. **TOML more complete than spec (reverse drift — important).**
   `task_edit`: TOML documents the editable content fields
   (`problem_statement`, `acceptance_criteria`, `context_required`,
   `constraints`, `handoff_output`); `actionSpecs` omits them.
   `forge` / `forge_edit`: TOML documents top-level `project`; `actionSpecs`
   omits it. So `actionSpecs` is **not** a clean superset today.

Only 4 of 42 actions agree on all overlap fields.

## Overlap-field inventory (what must be single-sourced)

The fields both systems carry, and which therefore can drift:

- **param name set** — the set of accepted keys (canonical + aliases).
- **per-param `required`** flag.
- **per-param `type`** (different vocabularies: spec `int64`/`object[]`/`string[]`/`bool`;
  TOML free-form `integer`/`optional_string`/`json object`/…).
- **per-param `description`** prose.
- **example** call payload.

Fields **only** actiondocs carries (its surface-specific prose — must be
**preserved**, not single-sourced): `purpose`, `[[param_aliases]]` structure,
`[[value_aliases]]`, `[[errors]]`, `notes`, `[[envelope_requirements]]` (e.g.
the `rationale` gate), `[returns]`.

## Decision

### Canonical source: `work.actionSpecs` (Go, in-code)

Rationale, in priority order:

1. **Flagless invariant (the constraint that decides it).** `actionSpecs` is
   always compiled in; actiondocs is gated on `--action-docs-dir`. The flagless
   consumers (`work_actions`, `CallShape`, `IdentifierRequiredError`) already
   read `actionSpecs` and must work without the corpus. Making the *gated*
   source canonical would invert the dependency (the always-present path would
   depend on a maybe-absent source). actiondocs as canonical is therefore ruled
   out.
2. **More correct on the dominant drift class.** On the ~17 identify-by actions
   the TOML is wrong (`slug` required, `id` missing); `actionSpecs` is right.
3. **Already carries examples** the TOML lacks on ~35 actions.

Caveat carried into T3: `actionSpecs` is **not yet a complete superset** (drift
class 5). T3 must first fold the params the TOML got right and the spec lacks
(`task_edit` content fields; `forge`/`forge_edit` `project`) **into**
`actionSpecs`, so the canonical source is complete before describe derives from it.

### Mechanism: code-canonical, generate the corpus from `actionSpecs` + `go:embed` it

> **Revised 2026-05-24** (author decision, supersedes the original
> runtime-derivation mechanism below). The Go catalog stays the single typed
> source of truth, but rather than overlaying it onto a hand-authored TOML at
> serve time, the **work corpus is generated from `actionSpecs` and embedded**.
> This kills the `--action-docs-dir` flag-gating entirely (the corpus is always
> present), keeps the existing `actiondocs.Registry` + docs-browser path intact
> (they read a real corpus), and makes describe agree with the catalog **by
> construction** because the corpus it serves *is* the catalog, materialized.
> The cost of the original runtime-derivation approach — coupling the generic
> describe handler to the work package, a bespoke type-vocab overlay, and
> splitting the Registry's population path — is avoided. The trade the author
> accepted: authoring moves fully into typed Go (param shape **and** the prose/
> structured fields), and the work `*.toml` files become generated artifacts.

End state:

- **`work.actionSpecs` (typed Go) is the complete source** for every work
  action's doc: param shape + example **and** the fields currently hand-authored
  in TOML (purpose, param aliases, value-aliases, errors, notes, envelope
  requirements, returns). Today most of these are structured data that port to
  Go cleanly; only `notes` is free prose (a backtick string).
- A **generator** emits `go/internal/actiondocs/corpus/work/*.toml` from
  `actionSpecs` (with a spec→doc type-vocabulary map), so the on-disk corpus
  stays the inspectable data file the docs browser reads. The corpus lives
  under the Go module (relocated from `blueprints/action-docs/` in T6) so
  `go:embed` can reach it.
- The corpus is **`go:embed`'d** into the binary → always present; the
  `--action-docs-dir`-gated "corpus not loaded" degraded path goes away (a dir
  override is kept for dev / hot-reload).
- `work_actions` + `CallShape`/`IdentifierRequiredError` read `actionSpecs`
  unchanged; `admin.action_describe` reads the embedded corpus. Both reduce to
  the same source.
- A **CI no-diff gate** (regenerate → assert clean) plus the repointed param
  gates (`param_tag_gate`, `param_type_parity`, now validating the canonical Go
  catalog for the work surface) make drift structurally impossible.

Task breakdown (revised): **T3** enrich `actionSpecs` to the complete, correct,
typed source (fold in `task_edit` content fields, `forge`/`forge_edit`
`project`, fix the `items`/`triggers` type errors and the `slug`-required /
`id`-missing class, reconcile `chain` vs `slug` canonical names; add `AliasOf`,
value-aliases, errors, notes, envelope-requirements, returns). **T4** write the
generator + regenerate the work corpus. **T5** `go:embed` the corpus + drop the
flag-gating. **T6** CI no-diff gate + repoint the param gates + un-skip the T2
parity gate. **T7** smoke (describe == CallShape == work_actions incl
`task_reopen`; flagless stdio serves describe) + retrospective + close.

#### Original mechanism (superseded): runtime derivation in `action_describe`

The first-pass plan overlaid `actionSpecs` onto the hand-authored TOML at serve
time (describe projects `Params`/`ParamAliases`/`Examples` from the catalog;
TOML keeps the prose). Rejected in favor of generate+embed because embedding
removes the flag-gating that motivated keeping prose in TOML, and a generated
corpus needs no runtime coupling between `admin` and `work`.

### Parity-test contract (for T2)

Define "agree" on the **observable `action_describe(work, X)` output** vs
`actionSpecs(X)` (not raw TOML — so the assertion survives T3 stripping the TOML
params). For every action in both sources:

- accepted-key **set** equal: describe `params[].name ∪ param_aliases[].from`
  == `actionSpecs` param names.
- `required` agrees on canonical params.
- type **family** agrees on shared params (`string`/`integer`/`bool`/`list`/`object`),
  tolerating vocabulary differences (`int64`≈`integer`).
- `example` present in both.

T2 exercises the describe handler (load corpus + `HandleActionDescribe`) so that
pre-T3 it **fails** on the 38/42 drift above, and T3 makes it pass by
construction.
