# Event Catalog

> **Status:** Initial registration set, produced by chain `agent-first-substrate` T1. Sibling to `docs/EVENT_SUBSTRATE.md` (envelope + mechanics) and `blueprints/events/*.json` (per-type payload schemas).
>
> **How to read:** one row per registered event type. Adding a new type means adding a row here, a schema in `blueprints/events/`, and wiring the emit-helper. See `EVENT_SUBSTRATE.md` §3.2.
>
> **Type-naming convention:** PascalCase, past-tense verb describing the state change (`BugResolved`, not `ResolveBug`). The past-tense voice matches the event-sourcing convention: events record what *happened*, not commands to do something.
>
> **Closed catalog.** Emitting an unregistered `type` is rejected at the dispatcher with `InvalidInput("unknown event type: <type>")` before any DB write. Agents cannot extend this catalog at runtime.

## Intentionally non-emitting actions

These mutating actions do NOT emit events in this chain. Each entry names the action and the rationale; readers folding the events ledger should not expect a row for these operations.

| Action | Reason |
|---|---|
| `forge_delete` | Per `EVENT_SUBSTRATE.md` §8.1 row 6: hard deletes on the existing CRUD tables are intentional (per `CONVENTIONS.md §Polymorphic-ref`) and *will-not-address-this-chain*. The closing event for each entity (`BugResolved` / `TaskCancelled` / `ChainClosed`) captures the lifecycle end; a separate "deleted" event would double-record the close. If a future chain retires the CRUD tables (Phase 4 of the audit), a `*Deleted` event family lands then. The events table itself is append-only by trigger — see `EVENT_SUBSTRATE.md` §3.4. |
| `task_block(blocker_slug)` without a status change | When `task_block` is called on a task already in `blocked` status to add an additional blocker (without changing the status), no `TaskTransitioned` event fires — the blocker-set update is row-only metadata on `task_blockers`. A future `BlockerAdded` / `BlockerRemoved` event family could capture this; not in this chain's scope. |
| `cleanupBlockersAfterClose` cascade unblock | When closing a task that was blocking others (`task_complete` / `task_cancel`), the `cleanupBlockersAfterClose` helper flips its dependents from `blocked` back to `pending` via a direct SQL UPDATE inside the same tx. This cascade transition is not currently emitted as a `TaskTransitioned` event. A faithful cascade emit would write each dependent's `TaskTransitioned(blocked → pending, blocker_slug = nil)` with `caused_by_event_id` pointing at the parent close event; deferred to a follow-up because the cascade is a side-effect of an already-emitted parent event and projection folds can derive the unblock from the parent's effect on `task_blockers`. |

## Conventions

- **Type names** are PascalCase, past tense, entity-prefixed where the entity is unambiguous (`BugResolved`, not just `Resolved`).
- **Payload field names** are snake_case to match the JSON-shape convention already used by the work meta-tool (e.g. `routed_chain_slug`, `commit_sha`).
- **Optional vs required.** A field is required if every emit-site must populate it. Optional fields are written as `field_name?` in this catalog and as `"required": false` in the JSON Schema. There is no allowed-but-deprecated form; deprecate by renaming the type (`BugResolvedV2`).
- **Cross-entity references.** The primary entity is in `entity`; secondary entities go in envelope `refs.related_entities`, **not** in the payload. Example: `BugResolved` with `kind="routed"` puts the bug in `entity` and `{kind: "task", slug: ..., project_id: ...}` in `refs.related_entities`. The payload's `routed_task_slug` is redundant with the ref but kept because it's part of the bug-resolve operational shape.
- **Project scoping.** Payload never carries `project_id` directly — it's in the envelope's `entity.project_id`. The catalog assumes the envelope is present; payload summaries only list payload-specific fields.
